*******************    CSFR    *******************

Navigato su www.example32.com, visualizzato i cookie che sono stati settati e attributo SameSite (sfrutto il fatto che il browser sa da dove parte la richiesta):
None - invia cross site
Lax - invia cross site SOLO verso dominio originale e SE user NAVIGA VOLONTARIAMENTE (es. link, invece image malevola con codice viene bloccata)
Strict - ONLY same site

NB: può aiutare a prevenire CSFR vedendo che ad esempio manca un cookie sul quale era stato disabilitato il cross-site

PREVISIONI:
 - Link A (same site): inviati tutti
 - Link B (da attacker32 a example32, link): inviato cookie-normal, cookie-lax
 - Link C (da attacker32 a example32, GET): inviato cookie-normal, cookie-lax
 - Link D (da attacker32 a example32, POST): inviato solo cookie-normal

TEST:
 - Link A
tutti i cookie sono stati inviati correttamente, questo per ogni possibile richiesta all'interno dello stesso dominio (es. anche POST)

 - Link B
ONLY cookie normal e lax, dato che l'utente ha intenzionalmente cliccato sul link

- Link C
ONLY cookie normal e lax, ANCHE SE é UNA GET malevola, perché teoricamente dovrebbero essere IDEMPOTENTI e NON MODIFICARE NULLA MA:

If a lazy developer creates a state-changing action using a GET request—for example, a link like http://good-site.com/transfer?amount=1000&to=hacker—CSRF defenses that rely on checking the Origin header will completely fail. 
Inoltre avrà accesso ai lax cookie.

- Link D
ONLY cookie normal, inoltre viene aggiunto il campo Origin negli header dato che POST  è un operazione che modifica una risorsa, così che server possa vedere da dove arriva la richiesta (in genere il campo referrer viene rimosso dai browser di default per questioni di privacy)
